3. 发布与标签

  1. 核心发布

    • 在 release/X.Y.Z 上完成最后测试→git merge --no-ff release/X.Y.Z → main

4. 分支更新策略

  • 核心→客户
    每当核心有新特性或安全修复落地 main，按需将 main 合并进各 customer/… 分支，确保客户版同步基础更新。

  • 客户→核心
    如果某客户的改动需要回馈到通用产品，请在客户分支上完成后，发 PR 到 develop 或 feature/… 分支，并经过评审后再合并。

  • 热修复流程

    1. 在 main 上创建 hotfix/… 分支
    2. 修复完毕后，merge → main 并打热修复标签
    3. merge → develop 保证开发分支同步

5. 小贴士

  • 分支生命周期
    特性分支、热修复分支寿命要短，完成即删 (git branch -d)；客户分支可长期保留。

  • 标签（Tag）管理
    统一以 vX.Y.Z 表示核心版本，以 <客户名>-vX.Y.Z 表示客户版本，方便回溯与 CI/CD 发布。

  • CI/CD 集成
    可在 main 分支打 tag 时自动触发核心构建发布；在 customer/* 分支打客户专版 tag 时触发对应客户环境部署。

通过上述规范，你可以既保持核心产品的迭代健康，又能灵活管理各客户的专版需求。若有更复杂的场景（如多级定制、白标化等），也可在此基础上细化分支层级或加入子模块／monorepo 结构。
